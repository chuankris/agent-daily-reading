# 05 RAG 的第一性原理

原文来源：

- Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks
- 链接：https://arxiv.org/abs/2005.11401

为什么今天读这个：

很多人一说 RAG 就只想到“向量库”。这篇论文能把你拉回原点：RAG 解决的到底是什么问题。

中文精读：

这篇经典论文的出发点很直接：

1. 纯参数化模型虽然存了很多知识
2. 但它访问知识、更新知识、给出可追溯证据的能力有限
3. 所以在知识密集型任务上，仅靠模型参数并不稳

RAG 的核心思路是把知识分成两类：

1. Parametric memory
- 模型参数中已经学到的知识

2. Non-parametric memory
- 外部可检索知识源，例如向量索引里的文档

真正重要的不是术语，而是工程含义：

1. 模型不需要把一切事实都“背下来”
2. 它可以在生成前或生成过程中先去查
3. 查到的内容还能给回答提供依据和可更新性

为什么这对工程落地重要：

1. 你可以更新知识库，不一定非要重训模型
2. 你可以要求回答带引用
3. 你可以把问题拆成“检索质量”和“生成质量”分别优化

今天先不用深挖公式，你要先建立的是这个认知：

RAG 不是一个向量数据库产品名，而是一种“让模型和外部知识协作”的系统设计。

你只需要记住的 3 个点：

1. RAG 是用外部知识补模型记忆的系统方法。
2. 它解决的不只是正确率，还有知识更新和可追溯性。
3. 检索和生成应分开看、分开评估。

和你的转型关系：

你未来非常适合做企业知识智能体。RAG 正是把“企业内部文档、接口说明、项目经验”接到模型能力上的桥。

今晚可动手：

用你熟悉的业务举 3 个例子，判断哪些问题必须依赖外部知识，而不能只靠模型“记忆”回答。
