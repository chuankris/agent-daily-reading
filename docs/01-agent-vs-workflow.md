# 01 Agent 和 Workflow 的区别

原文来源：

- Anthropic, Building Effective Agents
- 链接：https://www.anthropic.com/engineering/building-effective-agents
- 发布时间：2024-12-19

为什么今天读这个：

你后面学 MCP、RAG、LangGraph、OpenAI Agents SDK，如果这个基础边界没立住，很容易把所有东西都叫 Agent。

中文精读：

这篇文章最重要的贡献，不是教你某个框架，而是先把“带一点自动化的 LLM 应用”和“真正的 Agent”分开。

Anthropic 的区分很实用：

1. Workflow
- 路径由代码预先定义
- LLM 只是流程里的一个环节
- 适合边界清晰、步骤稳定的任务

2. Agent
- LLM 不只是生成文本，而是动态决定下一步做什么
- 它会选择工具、决定顺序、根据结果调整后续动作
- 适合开放性更强、步骤不完全固定的任务

对工程实践来说，这意味着：

1. 不要一上来就做 Agent
- 复杂度、成本、延迟都会上升
- 能用固定工作流解决的事，先别放权给模型

2. Agent 最适合“对话 + 行动”并存的任务
- 例如客服
- 例如代码修复
- 因为这些任务有反馈回路，也有相对可验证的成功标准

3. 工具设计的重要性不低于 Prompt
- 工具参数名、描述、输入格式，都会直接影响调用质量
- 很多失败不是模型太弱，而是工具定义让模型容易犯错

你只需要记住的 3 个点：

1. Workflow 是预定义路径，Agent 是动态决策路径。
2. Agent 不是越多越好，先用最简单能跑通的方案。
3. 生产里的 Agent 成败，很多时候取决于工具设计和反馈闭环。

和你的转型关系：

这篇特别适合你，因为你本来就做系统集成。你天然擅长画清楚流程边界。后面你要做的，不是抛弃这项能力，而是在“哪些步骤由代码固定，哪些步骤交给模型”之间做更聪明的划分。
