# 03 MCP 到底解决什么问题

原文来源：

- Model Context Protocol Specification
- 链接：https://modelcontextprotocol.io/specification/draft

为什么今天读这个：

MCP 是你后面面试和工程里都会反复碰到的概念。它比“单个工具调用”更偏统一接入协议。

中文精读：

MCP 的出发点很现实：如果每个 AI 应用都用自己的方式连接数据库、文件、API、IDE、知识源，生态会非常碎，工具也很难复用。

MCP 想做的是一层标准化协议，让 LLM 应用和外部上下文源之间可以更统一地交互。

从规范里看，MCP 里的核心能力可以先记成两组：

服务端常见能力：

1. Resources
- 给模型或用户提供可读取的上下文和数据

2. Prompts
- 提供模板化消息或工作流入口

3. Tools
- 提供可执行的函数能力

客户端常见能力：

1. Sampling
- 允许服务端请求进一步的模型交互

2. Roots
- 让服务端了解可操作的 URI 或文件系统边界

3. Elicitation
- 在需要时向用户追问额外信息

MCP 和普通 Function Calling 的差别可以先这样理解：

1. Function Calling 重点在“这次怎么调一个函数”
2. MCP 重点在“如何把工具、资源、提示模板、权限边界，用统一协议接进来”

规范里还有一个非常重要的点：安全。

因为 MCP 可能涉及数据访问和代码执行，所以用户同意、数据边界、工具安全、日志与取消机制都不是附属品，而是协议设计的一部分。

你只需要记住的 3 个点：

1. MCP 不只是工具调用，它是统一上下文和能力接入协议。
2. MCP 同时关心资源、工具、提示模板和交互边界。
3. 安全、授权、可控性是 MCP 的一等公民。

和你的转型关系：

你过去做系统集成，最熟悉的就是“异构系统怎么对接”。MCP 可以理解成 Agent 世界里的通用接入层思维，这和你原本的经验非常搭。
