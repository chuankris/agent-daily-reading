# 06 ReAct 为什么重要

原文来源：

- ReAct: Synergizing Reasoning and Acting in Language Models
- 链接：https://arxiv.org/abs/2210.03629

为什么今天读这个：

很多 Agent 系统的最基础执行范式，都能看到 ReAct 的影子。

中文精读：

ReAct 的名字来自两件事的结合：

1. Reasoning
- 先进行语言推理，明确当前判断

2. Acting
- 再采取动作，例如搜索、查询、执行工具

它想解决的问题是：

如果模型只会“想”，容易空想、幻觉；
如果模型只会“做”，又可能盲目行动、缺少全局判断。

所以 ReAct 让模型在一个循环里工作：

1. Thought
- 当前怎么理解问题

2. Action
- 下一步调用什么动作

3. Observation
- 动作执行结果是什么

4. 再进入下一轮 Thought

这个范式的工程价值在于：

1. 比单轮回答更适合复杂任务
2. 能利用工具返回结果纠正前一步判断
3. 轨迹天然更适合回放和调试

但它也有明显代价：

1. 步数变多
2. 延迟和成本上升
3. 如果缺少终止条件，容易死循环

所以生产里常见做法不是“盲目 ReAct”，而是：

1. 加最大步数
2. 加重复动作检测
3. 加验证器
4. 必要时和 Plan-and-Execute 混用

你只需要记住的 3 个点：

1. ReAct 是“边想边做”的闭环，不是只会输出答案。
2. 它把复杂任务拆成多步，并通过观察结果持续修正。
3. 它提升能力的同时，也提高了成本和治理要求。

和你的转型关系：

你以后做任何多步 Agent，都可以先用 ReAct 这个心智模型理解：它为什么会成功，为什么会漂移，为什么需要轨迹和回放。

用自然语言写一个 4 步以内的 ReAct 轨迹，主题可以是：

- 查询知识库后生成工单处理建议
