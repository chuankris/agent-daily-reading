# 09 为什么 Agent 一定要做评测

原文来源：

- OpenAI Agent Evals
- 链接：https://platform.openai.com/docs/guides/agent-evals

为什么今天读这个：

从这一篇开始，你的阅读会逐步从“概念理解”进入“工程质量”。Agent 和普通接口最大的差别之一，就是它很容易“看起来差不多，实际上质量已经退化了”。

中文精读：

这份官方文档最重要的提醒是：

1. Agent 质量不能只靠主观感觉判断
2. 必须用可重复的评测来观察变化
3. 评测不只是离线学术动作，而是工程迭代闭环的一部分

文档里反复强调几个关键对象：

1. Traces
- 先看运行轨迹，定位是 workflow 哪一步出了问题

2. Graders
- 用规则、模型或人工方式给结果打分

3. Datasets
- 维护一组可重复执行的样本集

4. Eval runs
- 用批量跑的方式比较版本变化

对你来说，最重要的不是 API 细节，而是要建立这个认知：

1. Prompt 改了可能退化
2. Tool schema 改了可能退化
3. 检索策略改了可能退化
4. 模型切换了也可能退化

如果没有评测，你根本不知道到底是变好了，还是只是“碰巧这次回答顺眼”。

所以 Agent 项目里的评测至少要回答三类问题：

1. 是否做对了
- 任务完成率、答案正确率、引用一致性

2. 是否做稳了
- 回归是否退化、错误分布有没有变化

3. 是否做贵了
- 延迟、token、工具调用次数、整体成本

你只需要记住的 3 个点：

1. Agent 系统必须有可重复评测，不能只凭体感迭代。
2. 评测对象不只是最终答案，还包括轨迹和中间步骤。
3. 评测的意义是支撑版本比较、回归发现和持续改进。

和 Agent 工程的关系：

你以后做项目时，真正能拉开和“只会调 API 的人”差距的，就是你能不能拿出一套最小评测集，证明自己在做工程，而不是做演示。

今晚可动手：

为你未来的主项目先想 5 条评测样本，写成表格：

- 输入任务
- 期望结果
- 是否必须用工具
- 你打算怎么判断成功
