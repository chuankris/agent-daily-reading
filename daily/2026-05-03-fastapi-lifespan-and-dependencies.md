## why read today

你前面已经补过 `pyproject.toml`、`pytest`、`asyncio`、MCP、Eval 和 tracing，下一步很自然会碰到一个更贴近真实工程的问题：不是“脚本能不能跑”，而是“Agent 服务怎么稳定地活着”。

对你这种做了 10 年 Java / Spring / IoT 集成 / ToB 交付的人来说，FastAPI 真正值得先补的不是路由语法，而是两件和工程骨架直接相关的东西：

1. 依赖注入：请求处理函数需要什么，就显式声明什么；
2. 生命周期：哪些资源该在应用启动时准备，哪些该在关闭时收掉。

这两件事如果心智不清楚，后面你做 Agent API、MCP 网关、工具服务、Eval 服务时，很容易把代码写成“全局变量 + 临时初始化 + 不知道何时清理”的脚本式结构。功能可能能跑，但一上并发、超时、重启、测试隔离，就会开始漏资源、状态混乱、难以维护。

## original-text translation

FastAPI 在 `Dependencies` 文档里先把核心定位讲得很直白：它提供了一套“强大但直观”的依赖注入系统。所谓依赖注入，就是你的路径处理函数可以声明自己运行所需要的东西，而框架会负责把这些依赖准备好并注入进去。官方特别强调，这种机制适合复用共享逻辑、共享数据库连接、施加认证和权限约束，同时尽量减少重复代码。

文档随后展示了最基本的形式：一个依赖本质上就是一个普通函数，它可以像路由函数一样声明参数；真正的路由函数通过 `Depends(...)` 标明自己依赖它。FastAPI 会自动解析这些依赖关系，先执行依赖，再把结果传入目标函数。官方还建议优先使用 `Annotated[..., Depends(...)]` 的写法，因为类型信息会保留得更完整，编辑器和类型检查工具也更容易理解。

在 `Lifespan Events` 里，官方把应用生命周期的边界说得很清楚：有些代码应该在应用真正开始接收请求前执行一次，也有些代码应该在应用关闭时执行一次。它覆盖的是整个应用的 lifespan。典型用途包括：初始化一个共享的数据库连接池、加载一份模型、准备跨请求复用的资源，以及在结束时清理这些资源。

FastAPI 推荐用 `FastAPI(lifespan=...)` 和异步上下文管理器来表达这种启动/关闭逻辑。也就是在 `yield` 之前做初始化，在 `yield` 之后做清理。文档还特别提醒，旧的 `startup` / `shutdown` 事件是替代方案，但如果提供了 `lifespan` 参数，对应的事件处理器就不会再被调用；换句话说，要么统一用 `lifespan`，要么统一用旧事件，不要混着理解。

如果把这两份官方文档合起来看，它想表达的工程观点其实很明确：请求级依赖和应用级资源是两层不同的问题，应该分别用依赖注入和生命周期机制来管理，而不是都塞进全局初始化或路由函数内部。

## Chinese deep summary

今天这两篇文档最值得你读的，不是因为“FastAPI 很流行”，而是因为它们正好解决你从 Java/Spring 世界迁到 Python/Agent 世界时最容易写乱的两条线：对象怎么装配，资源怎么托管。

先说 `dependencies`。如果只看表面，它像是“FastAPI 的参数魔法”；但从工程视角看，它更像请求处理层的轻量 DI 容器。你在 Spring 里早就熟悉一个思路：Controller 不应该自己到处 new 对象、拼配置、拉连接，而是声明自己需要什么，再由框架和配置层把对象装配好。FastAPI 的 `Depends` 也是同一个方向，只是形式更轻。

这点为什么对 Agent 工程尤其重要？因为你后面写的服务端接口，很可能并不是一个单纯的 CRUD handler，而是会有一串明确依赖：

1. 模型配置或租户配置；
2. 工具注册表或 MCP client；
3. trace / logger / usage recorder；
4. 鉴权信息、请求上下文、实验开关；
5. 某个 Agent runtime 或 orchestrator。

如果没有依赖注入意识，最常见的退化路径就是：路由函数里直接读环境变量，直接构造 client，直接去拿全局单例，直接在业务逻辑里掺认证和 tracing。短期看写得快，长期看会出现三个问题。第一，测试很难做，因为依赖不显式。第二，复用很差，因为同样的准备逻辑散落在多个入口。第三，边界不清，因为你分不出“请求自己的数据”和“整个服务共享的资源”。

FastAPI 的依赖系统之所以适合你，不是因为它和 Spring 一模一样，而是因为它保留了你最熟悉的那部分工程思想：依赖显式声明、逻辑可组合、共享能力可以抽出来复用。不同点在于，Spring 更偏容器式和配置驱动，FastAPI 更偏 Python 函数式组合。你不需要把它理解成“少了很多能力的 Spring”，更准确的理解是：它把很多常见服务装配问题，压缩成了更轻、更贴近语言本身的形式。

再说 `lifespan`。这部分几乎就是给 Agent 服务补“运行时骨架”。很多刚转 Python 的工程师，会不自觉把初始化逻辑放到模块顶层，或者塞进第一次请求处理时懒加载。前者的问题是：导入模块就有副作用，测试会慢，启动顺序不好控；后者的问题是：首次请求抖动大，资源初始化失败时错误边界很脏，而且关闭时常常忘了收尾。

`lifespan` 解决的是这类资源生命周期归属问题。哪些东西属于“整个应用共享一份”，就放到 `lifespan` 里管理。对 Agent 服务来说，典型例子非常多：

1. 启动时创建模型客户端、HTTP session、数据库连接池；
2. 预热一个向量检索索引或工具元数据缓存；
3. 初始化 MCP server 子进程管理器或长连接 transport；
4. 注册全局 tracing / metrics exporter；
5. 关闭时统一 flush 队列、关闭连接、释放句柄。

这和你原来做 ToB 系统交付时强调的“启动前准备好基础设施，停机时把资源干净释放”是完全同一类问题。区别只是现在这些资源更多和模型、工具、异步 IO、子进程、网络调用有关。Agent 工程里最怕的不是写不出一个 demo，而是 demo 变服务后，资源管理仍停留在脚本思维。

还有一个很重要的认知分层：`dependencies` 解决“这次请求要拿到什么”，`lifespan` 解决“整个应用活着期间共享什么”。这个边界一旦建立，很多设计会一下子顺起来。

比如一个 Agent API：

1. `lifespan` 里创建共享的 OpenAI client、MCP 连接池、数据库池；
2. 依赖函数里按请求提取租户配置、鉴权主体、trace context；
3. 再由一个更高层的依赖把这些东西组装成当前请求专属的 Agent runtime；
4. 路由函数自己只负责接收输入、调用 runtime、返回结果。

这套分层对你非常重要，因为它把“系统集成经验”直接翻译成了 Python Agent 工程的结构化写法。你原本擅长的是把复杂交付系统拆成边界清楚的层次，而不是把一切代码塞进控制器。FastAPI 的这两块机制，正好能承接这种长处。

最后还有一点经常被忽视：官方现在推荐 `lifespan` 而不是继续围绕 `startup` / `shutdown` 建新认知。这个细节看起来像 API 迁移，实质是在鼓励你用上下文管理器的方式看待应用资源边界。这和你昨天刚补的 `asyncio`、取消、清理语义是连上的。也就是说，FastAPI 这一层不是另起炉灶，而是在更高一层把 Python 的异步资源管理哲学落到了 Web/Agent 服务里。

## 3 key takeaways

1. FastAPI 的 `Depends` 本质上是请求处理层的依赖注入机制，适合把配置、鉴权、工具入口、runtime 装配从路由逻辑里拆出去。
2. `lifespan` 负责管理整个应用启动到关闭期间共享的资源，比把初始化写在模块顶层或散落在首个请求里更稳。
3. 对 Agent 服务来说，最关键的不是“会不会写接口”，而是先分清请求级依赖和应用级资源，这决定了后面的并发、测试、追踪和部署是否可控。

## relation to Agent engineering

这两篇文档和 Agent 工程的关系非常直接。

第一层是 Agent API 服务化。无论你后面是暴露一个聊天接口、一个工具执行接口，还是一个评测跑批入口，路由层都不该直接承担全部装配责任。`Depends` 可以把模型配置、用户上下文、鉴权、实验参数、trace context 这些请求级输入显式抽出来，让 handler 只处理“这一请求的业务编排”。

第二层是 MCP 和工具基础设施。一个 Agent 服务常常会共享 MCP client、HTTP client、连接池、缓存、工具注册表，这些都属于应用级资源，更适合放在 `lifespan` 中统一创建和释放。否则你要么每个请求都重建，浪费性能；要么偷放全局变量，边界含混。

第三层是测试与交付。你熟悉的企业交付经验会告诉你，最怕的是“本地能跑，启动顺序和关闭顺序没人说得清”。有了依赖注入和生命周期边界，测试时你更容易替换依赖、隔离副作用，也更容易把问题定位到“某个请求依赖没注入好”还是“某个全局资源没初始化或没释放干净”。

第四层是从 Spring 到 Python 的迁移。今天这篇不是让你忘掉 Spring，而是把你已有的 Controller-Service-Config-Bean 生命周期意识，迁到 FastAPI + Agent runtime 这套更轻的栈里。你的优势不是语法，而是早就知道大型系统为什么必须把装配和生命周期做清楚。

## a small action for tonight

今晚花 30 分钟做一个最小练习，不追求功能完整，只练工程分层：

1. 用 FastAPI 写一个只有一个接口的小服务，不要直接在路由里 new client。
2. 用 `lifespan` 在启动时创建一个共享对象，例如假想的 `tool_registry` 或 `llm_client`，关闭时打印清理日志。
3. 写一个 `Depends` 依赖，从请求头里解析一个 `tenant_id`，再写一个更高层依赖，把 `tenant_id` 和共享对象组装成一个简化版 `agent_runtime`，最后路由函数只调用这个 runtime。

如果今晚只记住一句话，就记这句：FastAPI 里最值得先学的不是“怎么接请求”，而是“请求依赖和应用资源分别归谁管”。
