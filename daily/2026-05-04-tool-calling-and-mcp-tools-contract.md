## why read today

你前 3 天刚补完 `pytest fixture / monkeypatch`、`asyncio / TaskGroup`、`FastAPI lifespan / dependencies`，这条线其实已经把 Python 服务骨架铺得差不多了。下一步如果还只停在“模型能不能调一下工具”，很容易又回到 Demo 心态，因为真正进入 Agent 工程后，难点不在“能调”，而在：

1. 工具定义是不是稳定契约，而不是一段松散描述；
2. 一次工具调用是不是完整闭环，而不是模型吐个参数你就当成功；
3. 错误、超时、权限、人机确认、输出结构这些边界是不是先想清楚。

对你这种做了 10 年 Java / Spring / IoT 集成 / ToB 交付的人来说，这一层特别值得早点补。因为 Tool Calling 本质上就是“模型参与的系统集成”，而 MCP tools 则把“工具如何被发现、调用、返回、报错”正式做成了协议。今天把这两份官方材料一起读，会比只读单边文档更有工程感。

## original-text translation

OpenAI 的 function calling 文档先把基本闭环定义得很明确：工具调用不是一次请求里自动完成的魔法，而是应用和模型之间的多步对话。高层流程包括五步：先把可用工具列表发给模型；模型返回一个或多个工具调用；应用侧执行对应代码；再把工具执行结果回传给模型；最后模型再生成最终回答，或者继续发更多工具调用。文档还强调，模型一次响应里可能包含零个、一个或多个调用，所以工程实现默认就应该按“可能有多个”来处理。

官方还提醒了两个很容易被低估的工程事实。第一，函数定义会被注入到模型上下文里，因此会占用上下文长度并计费；如果工具很多，应该控制描述长度、减少预加载工具数量，或者使用按需加载工具的方式。第二，工具真正交给模型的是 schema 契约，而不是你的内部代码对象；函数工具由 JSON schema 描述输入，模型据此生成参数。

MCP 的 tools 规范则把“工具”这件事进一步协议化。规范要求支持 tools 的服务端必须声明 `tools` capability；客户端通过 `tools/list` 发现工具，通过 `tools/call` 发起调用。每个工具定义至少要有唯一 `name`、人类可读的 `description`、以及 `inputSchema`；还可以带 `title`、`outputSchema` 和行为注解。规范还强调，工具的结果既可以是不结构化的 `content`，也可以带 `structuredContent`；如果声明了 `outputSchema`，服务端返回的结构化结果就必须满足这个 schema，客户端也应该做校验。

规范对错误和安全边界的表达也很直接。它区分两类错误：协议错误，比如未知工具、无效参数、服务端错误；以及工具执行错误，也就是协议层成功返回但业务执行失败，此时在结果里用 `isError: true` 表示。安全上，服务端必须校验输入、做访问控制、限流并清洗输出；客户端则应该在敏感操作前征求用户确认、向用户展示工具输入、校验结果并设置超时、记录审计日志。

## Chinese deep summary

如果把今天这两份材料压成一句话，那就是：在 Agent 工程里，工具不是“给模型多一个能力按钮”，而是“让模型进入受约束的系统集成回路”。

这点为什么适合你现在读？因为你已经不缺“系统之间有输入输出边界”这个意识了，缺的是把这个意识翻译成 Python / Agent / MCP 语境。很多刚做 Agent 的人，一开始只关心一件事: 模型会不会选对工具。但对有交付经验的人，更该优先关心的是后面那串问题：

1. 工具契约怎么定义；
2. 参数怎么校验；
3. 执行结果怎么回送；
4. 失败语义怎么分层；
5. 用户能不能看见、拦住高风险调用；
6. 一堆工具堆进去后，上下文和维护成本会不会爆。

OpenAI 文档最值得记住的，不是示例代码，而是“闭环”这个词。很多 Demo 看起来像一步完成，是因为它把中间几步藏掉了。但正式工程里，工具调用至少是双回合甚至多回合协议：

1. 你先暴露工具定义给模型；
2. 模型产出工具调用意图和参数；
3. 你的应用决定是否真的执行，并真正去调系统；
4. 你把结果再喂回模型；
5. 模型根据结果决定是继续调用还是给最终答复。

这和你以前做系统集成时的思路很像。外部系统不会替你自动把整条业务事务做完，它只会给出一个下一步需要的调用。你自己的编排层必须掌握控制权。Agent 工程里，模型也不该被误解成“自己会调用完工具并完成业务”，它只是参与了决策与参数生成。

这直接引出第一个关键工程判断：工具 schema 不是注释，而是接口契约。  
对 Java 背景的人，这个比喻最容易理解。你以前定义 DTO、OpenAPI、消息体 schema，不是为了文档好看，而是为了让上下游系统在边界上收紧。同样地，function calling 里的 JSON schema，和 MCP tools 里的 `inputSchema` / `outputSchema`，都不该被当作“给模型一点提示”。它们是：

1. 模型生成参数时的约束面；
2. 你自己做服务端参数校验的依据；
3. 评测工具调用稳定性的观测点；
4. 日后多人协作时最重要的共享边界。

如果 schema 松，后面的补救成本就会转移到解析、重试和脏数据清洗。比如一个“创建工单”工具，如果你只写一句 `create_ticket`，不给清晰字段定义、必填项、枚举值和附加字段限制，模型今天可能传 `priority=high`，明天传 `severity=P1`，后天又多塞一个你后端根本不认的字段。表面上看是“模型偶尔发挥不好”，本质上其实是接口契约没立住。

第二个关键判断，是要把“模型工具调用”与“协议工具调用”分层看。OpenAI 文档描述的是宿主应用和模型之间的闭环；MCP 规范描述的是客户端和服务端之间的协议。把两者拼起来，你才能看清真实 Agent 系统的层次：

1. 模型决定要不要调用某个工具；
2. 宿主应用决定是否允许、何时执行、如何记录；
3. 如果这个工具背后是 MCP server，再由 MCP 协议完成 `tools/list` / `tools/call` 的发现和执行；
4. 服务端返回结果或错误；
5. 宿主应用把结果整理后回送模型。

这个分层很重要，因为很多问题不是出在同一层。比如：

1. 模型选错工具，是模型决策层问题；
2. 参数缺字段，是 schema 或参数生成层问题；
3. MCP 返回 `Unknown tool`，是协议层或能力协商层问题；
4. 业务执行失败且 `isError: true`，是工具实现层问题；
5. 用户不该看到的敏感动作被执行了，是宿主应用的人机确认和权限边界问题。

这正是你以后做排障时最需要的思路。不要把所有失败都叫“Agent 不稳定”，而是先问：到底是哪个边界没收住。

MCP tools 规范里还有三个特别对工程味的点。

第一，`tools/list` 是显式发现，而不是默认存在。  
这说明工具集不是写死在模型脑子里，而是运行时能力面。对你来说，这很像系统集成里的能力发现或服务目录。工具能否被调用，不只是代码里有没有实现，还取决于当前连接、当前 server、当前 capability、甚至当前时刻工具列表是否变更。

第二，工具结果不只有一段文本。  
规范允许工具结果包含 `content`、`structuredContent`、资源链接、嵌入资源等，这意味着你后面做 Agent 时，不该把所有工具结果都压平成一段字符串。能结构化的，最好结构化；需要给模型和用户同时消费的，要考虑双重表示。这一点和你 4 月 27 日读的 Structured Outputs 是连上的，只不过今天它不再只是“模型最终答案要结构化”，而是“工具输出也应该尽量结构化”。

第三，错误必须分层。  
这点对 ToB 交付出身的人尤其重要。MCP 把协议错误和工具执行错误区分开，实际上是在帮你避免一个很常见的反模式：任何失败都只回一句 `tool failed`。正式系统里，这完全不够。你需要知道：

1. 是工具名根本不存在；
2. 还是参数非法；
3. 还是服务端内部炸了；
4. 还是业务执行失败但协议通了；
5. 还是用户主动取消了高风险动作。

这类分层一旦清楚，你后面做 tracing、eval、告警和重试策略才有基础。

还有一个不该忽视的点，是上下文成本。OpenAI 明说工具定义会占用上下文窗口并计费。这句话背后其实是在提醒你：工具不是越多越好。Java 背景的人很容易下意识想做一个“大而全工具平台”，把几十上百个动作都塞给模型。但到了模型上下文里，这意味着更长的描述、更高的歧义、更高的 token 成本、更低的选择精度。正确方向通常不是“先把所有工具都暴露”，而是：

1. 按场景裁剪工具集；
2. 把描述写短但边界写清；
3. 对低频工具做按需发现；
4. 对高风险工具加入确认门槛。

所以今天这篇真正要建立的，不是“我也会写一个 weather demo”，而是这套工程判断：

1. 工具定义是契约；
2. 工具调用是闭环；
3. MCP 让工具能力变成协议；
4. 安全、错误、超时、审计都不是补丁，而是设计的一部分。

## 3 key takeaways

1. Tool calling 不是模型一次输出就结束，而是“定义工具 -> 接收调用 -> 应用执行 -> 回送结果 -> 模型继续”的闭环。
2. `inputSchema` 和 `outputSchema` 应该被当成正式接口契约，而不是提示词附件；它们决定参数稳定性、错误定位和后续评测质量。
3. MCP tools 把工具发现、调用、结果结构、错误分层和安全边界协议化了，适合把 Agent 从 Demo 推向可交付系统。

## relation to Agent engineering

这篇内容和你后面的 Agent 工程几乎是一条直线关系。

第一层是工具设计。你以后无论接企业知识库、工单系统、设备接口、内部审批流，第一件事都不该是“怎么 prompt 模型去调”，而是先把工具 schema 设计好，明确必填字段、枚举、空值语义、错误返回和幂等边界。

第二层是宿主编排。真正的业务控制权应在应用侧，而不是交给模型自由发挥。你需要在模型提出工具调用之后，决定是否执行、如何鉴权、是否要求用户确认、超时多久、失败后是否重试、结果是否需要再清洗后回送。

第三层是 MCP 接入。MCP 很适合把企业内部能力做成标准化工具面，但它不是“接上就完”。你还要考虑 `tools/list` 的暴露范围、敏感工具的确认策略、`structuredContent` 的利用方式，以及协议错误和业务错误如何分别进 trace 与告警。

第四层是 Eval 和 Trace。以后如果模型经常选错工具、填错参数、滥调高风险动作，评测集和 tracing 里都应该能把问题定位到具体边界。否则你只会得到一句模糊结论: “这个 Agent 最近不好用了。”

## a small action for tonight

今晚花 30 分钟，不写完整 Agent，只做一个“工具契约草图”练习：

1. 选一个你熟悉的企业动作，例如“创建设备巡检工单”或“查询客户站点状态”。
2. 先写一个最小 `inputSchema`，明确必填字段、字段描述、枚举和 `additionalProperties` 策略。
3. 再写一个最小 `outputSchema`，不要只返回一段文本，至少把 `status`、`ticket_id`、`message` 这类结构化字段列出来。
4. 最后单独写 5 行备注：哪些失败属于协议层，哪些失败属于业务执行层，哪些调用必须人工确认。

如果今晚只记住一句话，就记这句：Agent 工程里的工具，不是“模型会用的函数”，而是“受协议、契约和宿主控制约束的系统接口”。

## 原文关键段落翻译（人工翻译，放在文末）

1. OpenAI 官方把工具调用定义为应用和模型之间的多步对话流程：先提供工具，再接收工具调用，执行应用侧代码，把结果回传模型，最后再得到最终回答或更多工具调用。
2. 官方明确提醒，模型响应里可能包含零个、一个或多个工具调用，因此实现时应默认按“可能有多个调用”来处理，而不是写死成单次调用流程。
3. 函数定义会占用模型上下文并按输入 token 计费，所以工具数量、字段描述长度和加载策略都应纳入工程权衡。
4. MCP 规范要求支持 tools 的服务端声明 `tools` capability，客户端通过 `tools/list` 发现工具，通过 `tools/call` 调用工具，每个工具至少要有名称、描述和输入 schema。
5. MCP 规范允许工具返回结构化与非结构化结果；如果声明了 `outputSchema`，服务端必须返回符合 schema 的结构化内容，客户端也应做验证。
6. MCP 把错误分成协议错误和工具执行错误两类，同时要求服务端做输入校验、访问控制、限流和输出清洗，客户端则应对敏感操作做人机确认、超时和审计。
