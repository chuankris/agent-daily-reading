## 为什么今天读这个
你这几天已经把 `uv`、`Structured Outputs`、`MCP 生命周期/传输` 补上了，接下来最容易踩坑的不是“再学一个框架”，而是避免自己掉进“Demo 能跑，生产不可控”的假进步里。  
对有 10 年 Java / Spring / IoT 集成 / ToB 交付背景的人来说，`Evals` 和 `Tracing` 其实不是新概念，它们分别对应你熟悉的“验收标准”和“链路追踪”。差别只是现在被测对象从接口服务，变成了会波动、会走工具、会做多步决策的 Agent 系统。  
如果这两层意识不先补牢，后面无论你写 Python Agent、接 MCP、还是把企业内服务包成工具，都很容易停留在“今天看起来不错”，却说不清为什么明天变差、差在什么环节、该怎么回归验证。

## 原文翻译（先读这一段）
OpenAI 在这两份官方文档里表达得很直接：第一，LLM 系统不能靠感觉验收，应该尽早定义评测目标、收集贴近真实分布的数据、设定可复用的评分方式，并把评测变成持续动作；第二，Agent 运行过程不该是黑箱，SDK 应该把一次工作流里的模型生成、工具调用、handoff、guardrail 等步骤都记录成可追踪的 trace 和 span。  
把这两句话翻成工程语言就是：你不能只盯着最终回答像不像对，而要同时建立“怎么测”和“怎么查”的能力。`Evals` 负责让你知道系统有没有退化，`Tracing` 负责让你知道系统退化时到底坏在哪一段。

## 中文深度总结
这两份文档最值得你现在补的，不是某个 API 细节，而是一种 Agent 工程观：`可观察` 和 `可评估` 不是上线后再补的配件，而是设计阶段就要一起落下去的主干能力。

先说 `Evals`。官方文档先把一个很多人会跳过的现实摆在台面上：生成式系统天然有波动，所以传统软件那种“固定输入，固定输出，断言等于某个值”的测试方式不够用了。这不代表不能测，而是要换成更贴合 LLM 特性的测法。文档建议先定义评测目标，再收集数据，再定义指标，再跑对比，最后持续评测。这个顺序很像你以前做 ToB 集成项目时的验收链路：先谈 SLA 和验收口径，再定测试样本和异常样本，再跑联调和回归，而不是先上线再靠客户现场反馈补洞。

这里面对你最重要的，不是“评测”这个词本身，而是它把 Agent 开发从“写提示词调手感”拉回到了“可重复比较的工程活动”。尤其文档里点名反对 `Vibe-based evals`，也就是“看起来好像能用”。这句话几乎可以当成你做 Agent 转型阶段的护栏。因为 Java 背景的人往往天然重视稳定性，但一到 LLM 项目里，周围环境会不断诱导你接受“差不多就行”的演示逻辑。官方这里其实是在提醒你：Prompt、工具、检索、模型、路由、上下文裁剪，只要任何一层变了，都应该有一组能复跑的 eval 去验证，而不是靠肉眼抽查。

第二个关键点，是数据集来源。文档推荐混合生产数据、历史日志、领域专家样本、人工标注样本，甚至必要时加入合成数据。这一点和传统后端最大的不同在于：Agent 的失败样本往往不在 happy path，而藏在边缘输入、歧义请求、异常工具返回、上下文缺口里。所以“日志就是以后最值钱的评测资产”这件事，要尽早建立意识。你以后真做企业 Agent，最先该存下来的不只是报错日志，而是“用户问了什么、模型怎么拆解、调用了什么工具、为什么答偏了、人工最后怎么修正”的全链路材料。没有这些，评测集永远长不出来。

再说 `Tracing`。Agents SDK 文档把 trace 和 span 讲得很清楚：一次端到端工作流是 trace，里面每个有开始和结束的步骤是 span。默认会追踪整次运行、每次 agent 执行、LLM generation、函数工具调用、guardrail、handoff，必要时还能加自定义处理器。对你来说，这套抽象并不陌生，它本质上就是把 Spring Cloud Sleuth / Zipkin / OpenTelemetry 那套链路思维，迁移到了 Agent runtime。

真正需要你建立的新理解，是 Agent 链路里的“失败点”比传统请求链更语义化。以前你查接口，主要看哪个服务超时、哪个 SQL 慢、哪个 HTTP 500。现在你还得查：模型在哪一步误判了意图、哪个工具参数构造错了、handoff 是否切错代理、guardrail 是否拦截了有效输出、上下文是否被压缩掉关键事实。也就是说，Agent tracing 不只是性能监控，而是“决策过程监控”。这正是它和普通日志的分水岭。

文档还有两个工程上很实用的提醒。第一，Tracing 默认开启，意味着官方认为它不是高级功能，而是默认基建。第二，某些 span 会带上敏感数据，比如 `generation_span()` 和 `function_span()` 里的输入输出，所以必须有意识地处理数据暴露边界。这一点对做过企业交付的人很关键，因为你应该立刻把它类比成生产日志脱敏、客户数据权限、审计留痕。Agent 能不能进真实业务，不只看能力，还看你有没有办法“既看得见问题，又不越过数据红线”。

把 `Evals` 和 `Tracing` 放在一起看，工程价值就更清楚了：`Evals` 解决“这次改动到底有没有让系统更好”，`Tracing` 解决“如果没有更好，具体坏在了哪一步”。前者像回归测试和验收基线，后者像分布式链路和问题定位。没有前者，你只能凭感觉迭代；没有后者，你即使发现分数下降，也不知道该改 Prompt、改工具定义、改检索、还是改状态流转。

对你当前的转型阶段，这比继续横向刷框架更重要。因为 Python、MCP、Agents SDK 都能很快学到“怎么接起来”，但真正决定你能不能从“会搭 Demo”升级到“会做可交付 Agent 系统”的，是你有没有从第一天开始就把评测和追踪一起放进工程闭环。

## 3 个关键收获
1. `Evals` 不是锦上添花，而是 Agent 开发的验收基线；要先定义目标、数据、指标，再谈优化。
2. `Tracing` 不只是日志增强，而是把模型生成、工具调用、handoff、guardrail 这些 Agent 决策步骤变成可追踪链路。
3. 真正能落地的 Agent 系统，必须同时具备“能持续测”和“出问题能定位”两种能力；只靠演示效果不算工程能力。

## 和 Agent 工程的关系
你后面无论走哪条路线，这两层都会直接复用：

- 做 Python Agent：每次改 prompt、工具 schema、上下文拼接方式，都应该挂一组最小 eval 集合。
- 接 MCP：不仅要测工具能不能调通，还要测工具失败、超时、返回脏数据时 Agent 的表现是否可接受。
- 做企业内交付：trace 要能回答“错在模型、错在工具、错在状态机、还是错在知识源”，否则现场排障会非常痛苦。
- 从 Java 迁到 Python：你以前熟悉的单测、集成测试、调用链、日志脱敏、验收口径，并没有过时，只是要映射到 Agent 语境里。

## 今晚可动手
花 20 分钟做一个最小练习，不写复杂业务，只搭一个“会调用 1 个工具”的小 Agent 草案，然后先不追求效果，先补两个文件：

1. 写一个 `eval-cases.jsonl`，只放 5 条样本，覆盖正常问法、歧义问法、工具失败、空结果、用户追问。
2. 再画一张你自己的 trace 草图：`user input -> agent reasoning/generation -> tool call -> tool result -> final answer`。
3. 最后问自己一句：如果第 4 条样本答错了，我现在是能看分数发现，还是只能靠肉眼撞见？如果发现了，我能不能定位错在工具还是错在提示词？

## 原文关键段落翻译（人工翻译，放在文末）
1. 官方评测最佳实践强调：不要把评测理解成只看一个 0 到 1 的分数。分数之外，还要结合人工判断，确认你测到的真的是业务关心的问题。
2. 官方给出的评测流程是：先定义评测目标，再收集数据集，再定义指标，然后运行和比较评测，最后把持续评测接进每次变更。
3. 文档明确反对“凭感觉评测”：不能因为“看起来像能用”就当成评测，更不能等系统上线后才第一次认真建立评测。
4. Agents SDK 的 tracing 文档说明：一次 Agent 工作流会被记录成 trace，里面的模型生成、工具调用、handoff、guardrail 等步骤会被记录成 span，便于在开发和生产中调试、可视化和监控。
5. 文档还提醒：某些 span 会捕获敏感输入输出，尤其是模型生成和函数调用，所以要根据场景控制是否记录敏感数据。
