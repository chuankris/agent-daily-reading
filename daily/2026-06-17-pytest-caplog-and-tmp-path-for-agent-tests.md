## why read today

你昨天刚补了 `pytest` 参数化和导入模式，这解决的是“样例怎么批量跑、回归到底测到哪份代码”两个基础问题。下一步如果还想把 Agent 工程继续做实，就不能只盯着最终回答文本了。

对 Agent 系统来说，很多真正昂贵的错误并不直接长在最后一句回复里，而是长在中间过程和外部副作用里，比如：

- 本该先澄清却没有澄清
- 工具调用失败了但没有打出足够日志
- 写文件写到了错误目录
- 临时产物没有清理，下一次运行被脏状态污染
- 明明走了 fallback，但测试只断言了“最后返回了字符串”

这正是今天这两篇官方文档值得读的原因。`caplog` 让你能把“运行时到底发生了什么”变成断言对象，`tmp_path` 让你能把“工具到底对文件系统做了什么”隔离在每个测试自己的沙盒里。对你这种做了 10 年 Java / Spring / IoT 集成、习惯按链路验收和交付口径控质量的人来说，它们的价值不是语法糖，而是把你原来很强的“过程证据 + 副作用边界”意识，迁移到 Python Agent 工程里。

## original-text translation

`How to manage logging` 这篇文档讲得很直接：`pytest` 不只是失败时顺带把日志打印出来，它还提供了 `caplog` fixture，让测试本身可以接管日志捕获和日志断言。你可以在测试里用 `caplog.set_level()` 或 `caplog.at_level()` 临时调高或调低日志级别，然后在运行结束后检查 `caplog.records`、`caplog.messages`、`caplog.text`、`caplog.record_tuples` 这些不同层次的结果。文档还特别提醒了一件很工程化的事：`caplog` 是通过往 root logger 上挂 handler 来工作的，如果测试过程中你用 `logging.config.dictConfig` 之类的方式把 root logger 直接重配掉，`caplog` 的 handler 可能会被抹掉，结果就是你以为自己在测日志，实际上什么也没捕获到。

文档里还有一个非常适合 Agent 测试的细节：`caplog.records` 默认只包含当前测试阶段的日志，如果你需要跨 `setup`、`call`、`teardown` 看完整证据，可以用 `caplog.get_records("setup" | "call" | "teardown")` 分阶段取。对 Agent 项目来说，这很像把一次运行拆成“准备上下文”“真正执行”“收尾清理”三段来验，能更精确地定位问题到底是初始化污染、执行路径偏移，还是清理逻辑失控。

`How to use temporary directories and files in tests` 则在补另一个工程缺口：文件系统副作用。文档说明 `tmp_path` 会为每个测试函数提供一个唯一的临时目录，而且它直接是 `pathlib.Path` 对象，不需要再在字符串路径和对象 API 之间来回切换。你可以在里面创建子目录、写文件、读文件、枚举目录内容，把“工具到底写了什么、写到哪、留下了几个文件”全都写成断言。文档还补充了 `tmp_path_factory`：如果某些磁盘资源生成成本高，可以做成 session 级别的共享临时目录，只生成一次，避免每个测试都重复造轮子。

另一个细节也很重要：`pytest` 默认会保留最近 3 次运行的临时目录。这个机制对排查 Agent 工具副作用问题很有帮助，因为当某条回归失败时，你往往需要回头看那次运行到底留下了什么文件树、什么中间产物、什么序列化内容，而不是只能看一句失败摘要。

## Chinese deep summary

如果把今天这两篇材料压成一句话，就是：

**Agent 回归要从“只验最终文本”升级到“同时验证日志证据和文件副作用”。**

第一层，你要开始把“日志”当成正式接口，而不是调试时顺手看的附属品。

很多刚转 Agent engineering 的人会下意识沿用一种很浅的测试方式：给输入，跑一下，看看最后字符串像不像预期。这种方式在简单问答上还能勉强成立，但一旦进到工具调用、澄清、重试、降级、审批、缓存、写文件、读外部数据这些真实工程路径，就会非常脆弱。因为真正的问题经常不是“最后一句错了”，而是“中间过程已经偏了，只是最后看起来还像回事”。  
`caplog` 的价值，就在于它逼你把这些中间证据显式化。比如你完全可以把下面这些东西写成测试断言：

- 缺少必要字段时，是否记录了一条 warning 并转入澄清
- 工具超时后，是否记录 retry 次数和退避信息
- 命中人工审批分支时，是否留下明确的 audit log
- fallback 到本地规则时，是否打印了足够上下文，便于回放

这件事和你以前做 ToB 集成验收很像。你真正信任一个链路，不是因为“最后页面有结果”，而是因为你知道每一步发生了什么，出了问题能顺着证据倒推。Agent 也是一样。日志不是附属品，它是决策链路的可观测证据。

第二层，`caplog` 不是为了看热闹，而是为了收紧行为合同。

一旦你开始断言日志，你就会倒逼自己把日志写得更像合同，而不是像情绪宣泄。什么地方必须打 info，什么地方必须打 warning，哪些字段必须出现，哪些异常必须带上下文，这些都会逐渐稳定下来。对 Agent 工程来说，这很关键，因为大量故障都发生在“模型和工具之间的接缝”里。  
如果日志结构模糊，你就很难区分：

- 是模型路由错了
- 还是工具 schema 不完整
- 是工具执行失败
- 还是失败被吞掉了

你原来 Java / Spring 集成工作里最强的一部分能力，其实不是“会接系统”，而是“知道出了问题怎么留痕、怎么定位、怎么复盘”。今天这篇正是在把这部分能力搬到 Python 里。

第三层，`tmp_path` 解决的是 Agent 项目里另一个特别容易脏掉的地方：文件系统副作用。

很多 Agent demo 一开始都喜欢直接读写本地文件，因为这样最容易打通。但一旦进入工程化阶段，如果你还让测试直接碰真实目录，就很快会遇到这些问题：

- 上一次运行留下的文件影响下一次测试
- 多个测试并发时互相踩目录
- 本该写到工作区缓存，却写到了项目根目录
- 序列化输出名字对了，但内容不对
- 工具声明“成功”，其实没生成任何有效产物

`tmp_path` 的工程价值就在于：它给每个测试一块独立、可检查、可丢弃的文件系统沙盒。这样你测试的就不再是“某个脚本似乎跑完了”，而是“这个 Agent / tool 在隔离目录里准确地产生了这些副作用”。这对 MCP tool、导出工具、代码生成器、知识库落盘、审计文件输出都特别关键。

第四层，`tmp_path_factory` 代表的是“测试资源分层”的意识。

不是所有磁盘资源都该每次现造。有些样本库、伪造文档、预生成知识片段、固定 schema 文件，其实更适合 session 级别复用。这个思路和你过去做系统集成时的“共享测试桩 / 固定基线数据”很像。区别只是现在用的是 pytest 的 fixture 体系，而不是手工脚本和环境约定。

第五层，把 `caplog` 和 `tmp_path` 放在一起看，你会发现它们恰好补齐了 Agent 回归里的两类证据：

- `caplog` 解决“运行过程中系统怎么想、怎么走、怎么报”
- `tmp_path` 解决“运行结束后系统留下了什么副作用”

前者偏可观察性，后者偏隔离和副作用验证。把这两层补上之后，你的测试才真正从“像 demo 的验证”走向“像交付系统的回归”。这一步对你尤其重要，因为它正好把你原来最擅长的集成交付思维，转成 Python Agent 工程里的硬能力。

## 3 key takeaways

1. `caplog` 的核心价值不是“把日志显示出来”，而是把 Agent 的中间行为证据变成可断言、可回归、可分阶段检查的测试对象。
2. `tmp_path` 的核心价值不是“随便给个临时目录”，而是把工具的文件系统副作用隔离进每个测试自己的沙盒里，避免脏状态和目录互踩。
3. 对 Agent 工程来说，可靠回归不能只验最终文本；至少还要能验运行日志和外部副作用，否则很多真实故障会被漏掉。

## relation to Agent engineering

这和 Agent engineering 的关系非常直接。

第一，它会改变你设计 Agent 测试的方式。你以后不该只断言“输出是否正确”，还该断言“有没有按预期留痕”“有没有按预期生成或修改外部产物”。这会直接提升 tool calling、MCP tool 封装、审批流、文件导出类场景的可测性。

第二，它会改变你写 Agent 代码的方式。只要你想用 `caplog` 测，就会自然推动你把日志字段、错误分类、重试信息、fallback 原因写清楚；只要你想用 `tmp_path` 测，就会推动你把路径注入、输出目录边界、产物命名规则设计清楚。换句话说，测试要求会反过来抬高实现质量。

第三，它特别适合你的背景迁移。你原来做 Java / Spring / IoT 集成时，本来就很重视链路日志、交付产物、问题复盘和环境隔离。今天要补的不是“质量意识”，而是用 Python / pytest 的方式把这种意识重新落地。`caplog` 和 `tmp_path` 正是这座桥。

## a small action for tonight

今晚做一个 40 分钟以内的小动作，不求搭大框架，只求把“日志断言 + 文件副作用断言”走通一次：

1. 选一个你最近最可能写的 Agent 工具场景，例如“读取输入文件并生成结构化摘要到输出目录”。
2. 用 `tmp_path` 给它准备一个输入文件和一个隔离输出目录，不要直接碰项目真实目录。
3. 在工具实现里只保留 3 类最关键日志：开始处理、关键分支、结束结果；然后用 `caplog` 断言这 3 类日志是否出现、级别是否合理。
4. 再对 `tmp_path` 下的产物做 3 个断言：文件是否生成、文件名是否符合规则、文件内容里是否包含你预期的关键字段。
5. 最后补一条自己的工程规则：以后凡是“会调用工具并产生副作用”的 Agent 能力，至少要有一条同时断言日志和产物的 pytest 回归样例。

## 原文关键段落翻译（人工翻译，放在文末）

1. `caplog` fixture 允许你在测试内部访问并控制日志捕获；你可以检查格式化后的消息文本、原始日志记录对象，以及适合直接做断言比较的 `(logger_name, level, message)` 元组。
2. `caplog` 提供 `set_level()` 和 `at_level()`，可以在测试期间临时调整日志阈值，并会在测试结束时自动恢复原来的级别。
3. `caplog.records` 默认只包含当前测试阶段的日志；如果你需要查看 `setup`、`call`、`teardown` 其中某一阶段的记录，可以显式使用 `get_records()` 按阶段读取。
4. `caplog` 通过向 root logger 添加 handler 来捕获日志；如果测试过程重新配置 root logger 并移除了现有 handler，日志捕获可能会失效。
5. `tmp_path` 会为每个测试函数提供一个唯一的临时目录，而且它直接是 `pathlib.Path` 对象，因此可以自然地用路径对象 API 创建目录、写文件、读文件和检查目录内容。
6. `tmp_path_factory` 是 session 级别的临时目录工厂，适合把生成成本高的测试资源只创建一次，再供多个测试复用。
7. `pytest` 默认会保留最近 3 次运行产生的临时目录，这对排查失败样例留下的中间文件和目录结构很有帮助。
