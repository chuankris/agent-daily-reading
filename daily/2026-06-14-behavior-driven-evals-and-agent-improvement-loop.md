## why read today

你这几天已经连续补了 `trace context`、`manual instrumentation`、`baggage`、`MCP tools contract`。这些内容解决的是“系统怎么接起来、怎么观测起来”。但如果再往前走一步，你很快会碰到一个更工程化的问题：

**看到问题，不等于能稳定修问题；修过一次，不等于下次不会回归。**

这正是 eval 的位置。对你这种做过 10 年 Java / Spring / IoT 集成、习惯按接口合同、回归测试、交付验收来思考的人来说，eval 最值得学的不是某个产品名词，而是下面这件事：

**怎么把“觉得 Agent 这次答得不错”升级成“我知道它为什么好、坏在哪、改完有没有真的变好”。**

今天这两篇资料刚好一内一外：

- `Working with evals` 讲的是 eval 应该怎么定义，尤其是怎么把模糊需求压成可评分的行为标准。
- `agent improvement loop` 讲的是拿到 traces 之后，怎样把失败样本、人类反馈、评估结果和代码修改串成真正的改进闭环。

这比继续追一篇“某模型又更强了”的资讯更适合你当前阶段，因为你现在缺的不是新名词，而是把 Agent 系统做成“可定位、可回归、可持续变好”的那套工程手感。

## original-text translation

`Working with evals` 这篇文档把“有用的 eval”拆成三个动作：先定义你到底要测什么，再准备能代表真实场景的数据，最后用合适的 grader 去判断系统表现。文档特别强调，eval 不应该只围绕“最终答案像不像标准答案”，而要尽量围绕具体行为来设计，比如是否遵守了指令、是否调用了正确工具、是否满足了某个明确约束。对复杂系统来说，好的 grader 往往不是“凭感觉打总分”，而是把行为拆成几个可检查的判断点，再组合成结果。

这篇文档还提醒了一个时间点：OpenAI 的旧 `Evals` 产品状态已经进入下线流程。文档写明，`2026-10-31` 会进入只读，`2026-11-30` 会完全关闭；如果你是现在重新建立评估习惯，应该优先转向 `Datasets`，但文档里关于“怎么定义 eval、怎么写 grader、怎么让评估贴近真实行为”的方法论仍然有效。

Cookbook 那篇 `Build an agent improvement loop with traces, evals, and codex` 则把方法论推进到工作流层。它不是只说“跑个 eval 看分数”，而是把流程写成一个闭环：先用 traces 识别失败运行，再让人类给出反馈或标注，再把这些样本沉淀成 eval 数据，再让系统或开发者基于结果修改 prompt、工具描述、路由或代码，最后重新评估。文档的重点不是某个 API，而是让你把“出问题了”变成“有证据地改、改完再验证”的日常机制。

把两篇放在一起看，最关键的一点是：

**eval 不是单独存在的一块测试面板，而是 Agent 工程里的质量控制回路。**

前一篇解决“怎么定义可测的好坏”，后一篇解决“定义完之后怎么推动系统持续变好”。

## Chinese deep summary

如果把今天这两篇材料压成一句话，就是：

**对 Agent 来说，真正有价值的 eval，不是答案评分，而是行为建模加改进闭环。**

第一层，要把“评估答案”升级成“评估行为”。

这是今天最值得你带走的认知。很多刚转 Agent engineering 的人，一说评估，脑子里默认还是传统问答题模式：给输入、看输出、判断像不像参考答案。这种办法在简单 FAQ 场景还凑合，但一旦进入 Agent，就很快不够用了。因为 Agent 的失误经常不是“最后一句话写错”，而是中间行为偏了，例如：

1. 该调用工具时没调。
2. 调了工具，但调错工具。
3. 工具选对了，但参数错了。
4. 工具结果拿到了，但没有按约束使用。
5. 明明应该升级、澄清或人工审批，却直接继续往下跑。

这些问题如果你只盯最终回答，很容易被掩盖；有时最后答案甚至看起来还行，但路径已经不可靠。`Working with evals` 的价值就在这里：它逼你把“好”翻译成一组可以检查的行为条件。比如，不要只问“回答是否正确”，而要补成：

- 是否选择了正确的工具。
- 是否在缺信息时先提澄清问题。
- 是否遵守了输出格式合同。
- 是否命中了安全或审批边界。
- 是否把关键字段从工具结果里正确提取出来。

这和你过去做 ToB 集成验收非常像。真正靠谱的验收从来不是“整体感觉不错”，而是把链路拆成若干明确检查点。Agent eval 本质上只是把这套方法移植到了模型驱动系统里。

第二层，grader 的核心不是“神判官”，而是“可复用裁判标准”。

很多人第一次接触 eval，会下意识希望有一个万能 grader，一眼看出系统好不好。这个想法通常不稳。文档给出的方向更务实：LLM grader 更擅长做有标准的判断、比较和分类，而不是无边界地给你产出完美终判。换句话说，你应该尽量把 grader 设计成下面这些形式：

- 判断某个约束是否满足。
- 比较 A/B 两个版本谁更符合要求。
- 对多个维度分别评分，而不是只给一个总印象。
- 先判断行为，再判断最终结果。

这对你尤其重要，因为你现在正在补 Python / Agent / MCP / Eval，而不是从零学工程思维。你真正该迁移过来的，是“先定义验收口径，再写实现”的习惯。grader 设计得越明确，后面不管你换 prompt、换模型、换工具实现，评估结果才越可比较。

第三层，traces 的价值不在“看热闹”，而在“产出可回归样本”。

Cookbook 那篇讲的改进闭环，最有工程味的地方就在这里。trace 不是终点。它真正的作用，是把一次失败运行里的上下文、调用路径、工具使用、系统决策保留下来，然后让你回答两个问题：

1. 这次失败到底坏在哪一层？
2. 这类失败以后怎么防止再发生？

如果你只是看 trace 然后手工改一下，这还是“救火模式”。只有当你把失败样本抽出来，配上人类反馈或评估标准，沉淀进数据集里，下次改完再自动或半自动重跑，你才开始进入“回归模式”。

这里特别贴合你原来的职业背景。你过去做 Java / IoT 集成时，现场问题如果只靠工程师记忆，下个客户还会再踩一遍。只有把问题归档成测试、脚本、验收规则或监控告警，团队才算真正学会。Agent 这边也一样：trace 负责发现问题，eval dataset 负责记住问题，grader 负责重复检查问题。

第四层，今天要补的是“评估设计能力”，不是“评估平台操作能力”。

文档明确给了一个现实日期：旧 `Evals` 产品会在 `2026-10-31` 只读，并在 `2026-11-30` 关闭。这个信息反而是好事，因为它提醒你别把注意力放错地方。你当前阶段最值得积累的，不是某个具体控制台怎么点，而是这些可迁移能力：

- 怎么从真实失败里抽样。
- 怎么定义行为型评分标准。
- 怎么区分“模型没做好”与“工具合同没设计好”。
- 怎么让每次修改后都有最小回归集。

只要这套能力在，你以后不管用 OpenAI 的 Datasets、自己写 pytest 驱动 eval、还是接别的评估平台，迁移成本都不高。反过来，如果只学会某个平台按钮，你会很快被产品生命周期带着走。

第五层，对你这种“Java/Spring 强，Python/Agent 还在补”的人，eval 是建立技术自信的最好抓手。

因为 eval 最不靠“会不会最新花活”，最靠“有没有把复杂系统拆清楚”。这恰好是你的长板。你现在补的 Python、MCP、trace、structured output，其实最终都会汇到同一个问题上：**系统改了以后，怎么证明它真的更好，而且没把别的地方搞坏。**

一旦你把这个闭环搭起来，你就不只是“会写一个 Agent demo”的人，而是在往“能维护 Agent 质量的工程师”走。这两者在岗位竞争力上差很多。

## 3 key takeaways

1. 对 Agent 系统来说，eval 最该先测的是行为而不是答案；工具选择、参数正确性、澄清、审批、约束遵守这些中间动作，往往比最终回答更能暴露真实问题。
2. 好的 grader 本质上是明确、可重复的裁判标准，不是万能主观打分器；优先设计成规则判断、A/B 比较、分维度评分这类稳定结构。
3. traces 只有转化成失败样本、反馈标注和最小回归集，才会真正变成工程资产；否则它只是一次性的排障记录。

## relation to Agent engineering

这和 Agent engineering 的关系非常直接，而且正好贴着你当前缺口。

第一，它会改变你调 Agent 的方式。你以后不该只问“这次结果对不对”，而要问“这次行为链哪一步开始偏了”。这会直接影响你怎么读 trace、怎么拆工具问题、怎么定位 prompt 和 routing 的职责边界。

第二，它会改变你写 Python 工程代码的方式。你现在补的 pytest、mock、结构化输出、JSON Schema，不应该孤立地学；它们最终都会服务于 eval。比如，结构化工具输出能让 grader 更稳定，pytest 参数化样例能让最小回归集更便宜，mock 能帮你隔离外部依赖，把失败锁定在 Agent 决策层。

第三，它会改变你桥接 Java 与 Python 的策略。未来很可能不是“把老系统全改成 Python”，而是 Python Agent 编排 + Java 业务能力继续沉在后面。那时 eval 要覆盖的就不只是模型文本质量，还包括跨系统工具合同、审批边界、失败恢复和可解释性。你现在学会行为型 eval，后面才能把这条链真正管住。

## a small action for tonight

今晚做一个 35 分钟以内的小动作，不求平台化，只求先把“行为型 eval”搭个最小样子：

1. 选你最近一个 Agent 场景，写出 5 条真实输入，其中至少 2 条是失败或边界样本。
2. 不要先写“标准答案”，先给每条样本写 3 个行为检查项，例如“是否先澄清”“是否调用正确工具”“是否遵守审批边界”。
3. 给每个检查项定义一个最简单的三值结果：`通过 / 部分通过 / 失败`。
4. 从其中 1 条失败样本回看 trace，标出第一处偏移发生在哪一步，并写一句“如果要避免回归，这条样本以后要怎么保留到数据集里”。
5. 最后补一条自己的工程规则：以后每次改 prompt、tool schema、routing 或 guardrail，都至少重跑这 5 条样本。

## 原文关键段落翻译（人工翻译，放在文末）

1. 一个有用的评估通常包含三步：先定义要测的内容，再收集数据，最后使用 grader 对系统表现做判断。
2. 对复杂 Agent 系统，评估应尽量围绕可观察行为来设计，而不是只围绕最终文本答案是否“像正确答案”。
3. LLM grader 更适合做有明确标准的判断、比较和分类，因此应尽量把评估问题压成更可判定的形式。
4. 旧 `Evals` 产品将于 2026 年 10 月 31 日进入只读状态，并在 2026 年 11 月 30 日关闭；新的评估工作应优先转向 `Datasets`。
5. Agent 改进闭环的关键不是只看一次失败，而是把 traces、人工反馈、eval 数据和代码修改连接起来，让系统可以持续迭代。
6. trace 的价值在于保留完整运行证据，而 eval 的价值在于把这些证据转成可重复检查的样本与标准。
