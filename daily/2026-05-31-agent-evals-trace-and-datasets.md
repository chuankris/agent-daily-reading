## why read today

你这轮补课到现在，已经把 Python 工程化、Tool Calling、MCP、Trace Context、OpenTelemetry 语义约定这些“系统能搭起来”的骨架补得差不多了。接下来最容易卡住的，不再是“能不能写出一个 Agent demo”，而是：

1. 改了 prompt、tool schema、路由逻辑之后，你没有稳定办法判断到底是变好了还是只是碰巧跑通。
2. 出现错误时，你只能盯最终答案，却看不清问题发生在选错工具、handoff 错误、上下文丢失，还是某一步 workflow 偏了。
3. 你明知道评估重要，但容易把 eval 理解成“拿几条 case 手工试试”，而不是工程里的持续机制。

这两篇官方文档刚好补这个短板。它们给出的不是一套炫技框架，而是一个非常工程化的判断顺序：

1. 先用 trace 看清行为。
2. 再把“什么叫好”沉淀成 dataset 和 grader。
3. 然后把评估做成持续运行的回归机制。

这对你这种做过 10 年 Java / Spring / IoT 集成 / ToB 交付的人特别重要。因为你的优势从来不是“最早学会一个新 SDK”，而是把复杂系统做成可验证、可回归、可上线、可排障的东西。Agent 工程走到后面，拼的也正是这个。

## original-text translation

`Evaluate agent workflows` 这篇文档先给了一个很实用的决策顺序：如果你还在排查 agent 行为、还没完全搞清楚系统到底为什么好或为什么坏，就先从 traces 开始。因为一条 trace 记录的是一次运行的端到端过程，里面会把模型调用、工具调用、guardrails、handoffs 这些关键动作串起来。文档建议先检查代表性的 trace，再用 grader 去给这些 trace 打结构化分数，这样你能更快定位 workflow 级的问题。

这篇文档接着说，当你已经逐步知道“什么样的行为算好”之后，就应该从单条 trace 观察，走向可重复的 datasets 和 eval runs。也就是说，评估不能一直停留在“翻日志看一眼”，而要进入能对比版本、能长期复跑、能量化回归的阶段。如果还需要更高级的能力，例如更大规模运行、通过 API 管理评估、或者评估外部模型，可以继续用 Evals API。

`Evaluation best practices` 则把评估流程拆成更完整的工程闭环。它建议先定义评估目标，再收集数据集，再定义评估指标，然后运行和比较评估，最后把持续评估建立起来。文档特别强调，随着系统从单轮调用升级到 workflow、单 agent、multi-agent，系统中的不确定性会越来越多，而 eval 的作用正是去找到这些不确定性落在哪一层。

这篇文档还有两个很值得记的提醒。第一，大模型更擅长做区分和比较，而不是替你生成一个完美标准答案，所以评估最好尽量落在 pairwise comparison、分类、按标准打分这类任务上。第二，数据集不能只放“正常样本”，还要包含 edge cases、adversarial cases、生产中出现过的失败样本，以及人工整理出的高价值样本。换句话说，好的 eval 不是多跑几次，而是让数据、标准和迭代节奏都进入工程体系。

## Chinese deep summary

如果把今天这两篇材料压成一句话，就是：

**Agent Evals 不是“最后验收一下答案”，而是把 Agent 系统的行为观察、质量标准和回归机制连成一个闭环。**

第一层，你要先把 eval 从“看最终答案”升级到“看完整执行过程”。

很多人一说评估，脑子里只有一个问题：最终回答对不对。这种方式对单轮问答勉强够用，但对 Agent 很快就不够了。因为 Agent 的失败，常常不是最终那句话本身出了问题，而是中间过程已经偏了。例如：

1. 工具选错了。
2. 该 handoff 时没有 handoff。
3. 工具参数拼错了。
4. guardrail 被绕开了。
5. workflow 某一步拿到了不完整上下文。

如果你只盯最后输出，你会看到“答错了”；但如果看 trace，你能看到“为什么答错”。这就是官方文档建议先从 traces 开始的原因。对当前阶段的你来说，这一步尤其关键，因为你还在把 Python、MCP、Agent runtime 的各层边界慢慢吃透，trace 是把这些边界真正照亮的工具。

第二层，trace 是诊断工具，dataset + eval run 才是回归工具。

这两个东西不能混。trace 更像你以前做系统联调时的单次现场抓包、调用链回放、故障定位；dataset 和 eval run 更像你把这些经验沉淀成测试集，然后每次改动都重新跑一遍。

很多刚转 Agent 工程的人会停在 trace 这一步，因为它最直观：看到一次失败，修一次。但只靠 trace，你没法稳定回答下面这些问题：

1. 这次改 prompt，是整体变好，还是只修好了一个案例？
2. 这次改 tool schema，会不会让另一个场景退化？
3. 这次改路由逻辑，对历史失败样本有没有回归？
4. 同一个 agent 在两周后是不是又把老问题带回来了？

所以官方文档的顺序很对：先用 trace 看清系统，再把“代表性问题”沉淀进 dataset，然后用 eval runs 周期性复跑。你以前做 Java/IoT 交付时，如果一个接口问题只在现场靠人记，很快就会重复踩坑；只有进回归集，团队才真的学会了。这在 Agent 里完全一样。

第三层，评估目标、数据集和指标必须分开想，不能混成一句“我想让效果更好”。

这点非常工程化，也很适合你。`Evaluation best practices` 给出的五步其实是在逼你把模糊目标拆开：

1. 目标到底是什么，是答得更准、路由更稳，还是工具调用更对？
2. 数据从哪里来，是生产日志、人工构造、历史失败样本，还是专家参考答案？
3. 指标是什么，是准确率、pairwise 偏好、context recall，还是结构化打分？
4. 比较对象是什么，是两个 prompt、两个模型、两个工具版本，还是两套路由？
5. 这套评估是一次性跑，还是每次改动都跑？

你过去做企业项目应该很熟悉：凡是“目标、样本、验收标准”混在一起的项目，最后一定容易扯不清。Agent Eval 本质上也是把这三件事拆开，然后让团队能持续对齐。

第四层，大模型评估更适合比较、分类和按标准打分，不适合把“开放生成”直接当唯一标准。

这是今天最值得直接拿回去用的一个判断。很多人做 eval 时，会下意识写成“请判断这个答案好不好”，然后希望 grader 给出万能结论。官方文档提醒得很对：LLM 更擅长做比较和判别，而不是凭空定义完美答案。因此更稳定的设计通常是：

1. 比较 A 和 B 哪个更符合标准。
2. 判断输出是否满足某条明确规则。
3. 按若干维度分别打分，而不是一句总评。
4. 对 tool calling 分别检查“工具名是否对”“参数是否对”。

这和你熟悉的系统验收很像。真正可靠的验收标准，通常不是“整体感觉不错”，而是把关键行为拆成几个可检查的点。Agent Evals 只是把这套思路搬到了 LLM 场景里。

第五层，好的 eval 数据集，核心不是“多”，而是“覆盖真实失败模式”。

对你现在的转型阶段，这一点尤其容易忽略。因为你可能先用几个干净样例跑通，然后误以为自己已经有评估了。实际上，真正有价值的数据集往往要故意放入：

1. 正常样本
2. 边界样本
3. 对抗样本
4. 历史失败样本
5. 来自生产日志的真实用户输入

为什么这和你有关？因为你未来做的很可能不是纯实验室里的玩具 agent，而是带企业接口、知识库、审批流、权限边界的系统。这样的系统最容易死在长尾场景里。评估数据集如果没有这些场景，系统上线后就会把你熟悉的 ToB 老问题重新演一遍。

第六层，持续评估不是锦上添花，而是 Agent 工程开始“像工程”的分水岭。

文档最后强调 continuous evaluation，这一点非常值钱。因为 Agent 系统天然有不确定性：模型会变、prompt 会变、tool 会变、数据源会变、路由会变、依赖服务会变。只做一次评估，意义有限；把评估接进每次改动后的流程里，才真正开始建立质量护栏。

对你来说，这也是把原有工程优势迁移过来的最好入口。你并不需要一开始就做很复杂的平台，先做一个最小闭环就够：

1. 从 trace 里挑失败样本。
2. 沉淀成 dataset。
3. 给每类问题定义 grader 或规则。
4. 每次改 prompt / tool / workflow 后重跑。
5. 观察回归，而不是只看演示是否成功。

这套方法一旦建立，你就不再只是“会做 Agent demo”，而是在往“会维护 Agent 系统质量”的方向走。

## 3 key takeaways

1. 对 Agent 系统来说，先看 trace 再做 datasets/eval runs 是正确顺序，因为单看最终答案无法定位 workflow、tool、handoff、guardrail 这些中间层问题。
2. 一个合格的 eval 闭环至少要拆开四件事：目标、数据集、指标、持续运行机制；否则评估很容易退化成零散试 case。
3. 评估设计要尽量利用 LLM 擅长比较和判别的特点，把复杂任务拆成可检查的规则、分类、分项打分或 tool 调用子项评分。

## relation to Agent engineering

这篇内容和 Agent engineering 的关系非常直接，而且正好贴着你当前的短板。

第一，它会改变你做 Agent 调试的方式。以后你不该再只问“回答对不对”，而要先问“trace 里哪一步开始偏了”。这会直接提升你排查 tool、MCP、workflow 问题的速度。

第二，它会改变你怎么设计测试。你过去熟悉的是接口测试、集成测试、回归测试；在 Agent 世界里，这套习惯要迁移成 datasets、graders、eval runs 和持续评估。底层方法论其实是相通的。

第三，它会帮助你把 Python 补课和工程习惯连起来。你现在学的 pytest、mock、结构化输出、JSON Schema、trace 语义，这些不是孤立技能，最后都会汇到“如何让 Agent 系统可验证”这个问题上。

第四，它也能帮你更好地桥接 Java 背景。很多真实项目完全可以把企业接口、规则引擎、审批链、设备侧能力继续放在 Java/Spring 里，而把 Agent 编排放在 Python。Eval 闭环则负责跨语言地衡量整个系统是否真的变好。

## a small action for tonight

今晚做一个 40 分钟的小动作，不求平台化，只求把 eval 闭环先搭出最小版本：

1. 从你最近做过的一个 agent / workflow 里挑 5 条真实输入，其中至少 2 条是失败样本。
2. 给这 5 条样本各写一句“这条样本最关心什么”，例如工具选择、参数正确性、最终答案是否满足约束。
3. 手工定义一个最小评分表，不求复杂，只要能区分：`通过 / 部分通过 / 失败`。
4. 对其中 1 条失败样本，把整条 trace 拆成 3 到 5 步，标出第一处偏移发生在哪一步。
5. 最后写下一个你自己的规则：以后每改一次 prompt、tool schema 或 routing，都至少重跑这 5 条样本。

你先把这个最小回归集做起来，后面无论接 OpenAI Evals、自己写 pytest 驱动 eval，还是接 CI，都会顺很多。

## 原文关键段落翻译（人工翻译，放在文末）

1. 当你还在调试 agent 行为时，应先从 traces 开始，因为 trace 能记录一次运行里模型调用、工具调用、guardrail 和 handoff 的完整过程，适合定位 workflow 级问题。
2. 一旦你逐渐知道“什么样的行为算好”，就应该从单条 trace 的观察，转向可重复的 datasets 和 eval runs，用它们长期比较版本变化和回归情况。
3. 一个完整的评估流程至少包含：明确目标、收集数据集、定义指标、运行并比较评估，以及建立持续评估机制。
4. 随着系统从单轮调用升级到 workflow、单 agent、multi-agent，不确定性会不断增加，因此评估要优先落在那些真正引入不确定性的层面。
5. 大模型更擅长做比较、分类和按标准打分，所以评估设计应尽量利用这些能力，而不是把开放式生成当成唯一评分方式。
6. 一个有价值的 eval 数据集应同时包含正常样本、边界样本、对抗样本、历史失败样本和来自真实生产的数据。
7. 持续评估的目标不是一次性打分，而是让每次 prompt、模型、工具或 workflow 变更后，都有办法发现回归和验证改进。
