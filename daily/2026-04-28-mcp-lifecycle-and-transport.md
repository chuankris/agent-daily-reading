为什么今天读这个：

你前两天已经补了 Python 工程化和 Structured Outputs，下一步不该继续停在“模型能调工具”的表层，而要开始理解 Agent 工程里真正像系统集成的部分：协议、握手、能力协商、会话边界、传输方式。

对你这种做过 10 年 Java/Spring/IoT 集成和 ToB 交付的人来说，MCP 真正值钱的地方，不是“又一个 AI 名词”，而是它把工具接入、资源暴露、权限边界、客户端能力、会话状态这些以前分散在各系统里的东西，收成了一套正式协议。如果这一层没补扎实，后面看 MCP SDK、Spring AI MCP、远程 MCP Server、安全边界时会一直停留在二手理解。

原文翻译（先读这一段）：

MCP 官方规范在这两篇里表达得很明确：它不是“列一个 tools 清单”就算完成，而是先定义客户端和服务端如何建立连接、协商协议版本、声明双方能力、进入正常运行阶段，再按约定的传输方式交换消息并结束连接。也就是说，MCP 的核心不是某个单独 API，而是一整条会话通路。

在生命周期上，规范要求初始化必须是客户端和服务端的第一轮交互。客户端先发 `initialize`，带上自己支持的协议版本、客户端能力和实现信息；服务端返回它支持的版本、服务端能力和实现信息；初始化成功后，客户端再发 `initialized`，之后双方才进入正常操作阶段。也就是说，MCP 不是上来就直接调工具，而是先做一轮正式握手。

规范还强调：后续通信只能使用双方已经协商成功的能力。比如服务端声明了 `tools`、`resources`、`prompts`，客户端没声明某些对应支持能力，那么会话里就不能假定这些能力一定可用。这个思想非常像你做系统对接时先谈接口版本、认证方式、字段约束，再谈业务调用。

中文精读：

这两份官方规范最值得你现在补的，不是 API 细节，而是“把 MCP 看成一条会话协议”。

很多人第一次听 MCP，会直接把它翻译成“给大模型接工具的标准”。这句话不算错，但太浅了。因为如果你只看到 tool，你会很容易把 MCP 误解成：

1. 一个工具注册表。
2. 一个函数调用格式。
3. 一个给模型读文件、调接口的小框架。

但官方规范的写法清楚说明，它更接近“面向 Agent 运行时的上下文交换协议”。工具只是其中一个原语，不是全部。协议里还有：

1. 生命周期管理：先初始化，再运行，再关闭。
2. 能力协商：客户端能给什么，服务端能提供什么。
3. 客户端能力：例如 roots、sampling 等。
4. 服务端能力：例如 tools、resources、prompts、logging。
5. 会话约束：后续只能在已协商能力内通信。

这套思路和你熟悉的企业集成非常像。你以前对接硬件、平台、第三方系统时，不会接受“先直接调接口，后面再说版本和权限”。MCP 也是一样。它先把“边界”放到前面。

这里最该重点理解的是初始化流程。规范要求 `initialize` 必须是第一轮交互，而且初始化未完成前，双方都不该乱发别的请求。这一点很像企业协议里的握手阶段，先确认：

1. 你是谁。
2. 你支持哪个版本。
3. 你会哪些能力。
4. 我能不能按这个版本和你对话。

为什么这一步很关键？因为 Agent 工程最大的坑之一，就是大家都默认“连上就能用”。一旦进入真实生产环境，这种假设会马上出问题：

1. 客户端和服务端支持的 MCP 版本不同。
2. 客户端以为服务端支持 `resources`，但实际上只支持 `tools`。
3. 服务端以为客户端能处理某类交互，但客户端根本没实现。
4. 会话状态丢失后，双方还以为自己在同一个上下文里。

你如果用系统集成经验来理解，这和设备协议协商、接口版本兼容、能力探测、会话保持完全是同一类问题。区别只是现在对接对象换成了 Agent runtime。

第二个很值钱的点，是“客户端能力”这件事。很多人习惯把服务端当成主角，觉得 MCP 就是 server 暴露工具给 client 用。但从官方规范看，客户端也不是被动壳子。客户端可以声明 `roots`、`sampling` 等能力，这意味着：

1. 客户端可以告诉服务端，哪些文件系统根目录是可见边界。
2. 某些能力并不是服务端单方面说了算，而是双边协商。
3. 用户控制、宿主应用控制、模型控制之间是有分层的。

这对你很重要，因为你后面做 MCP 时，不该只想“怎么暴露工具”，还要想“宿主侧要不要给这个 server 文件根目录”“要不要允许它请求更多上下文”“边界变化时怎样通知”。这正是生产级 Agent 和 demo 的分水岭。

第三个关键点，是 MCP 不是天然只适合本地 `stdio`。官方规范把传输层也规范化了，说明它既能跑本地子进程模式，也能跑 HTTP 会话模式。对转型中的你，这个区分非常重要：

1. `stdio` 更像本地工具进程接入。
2. HTTP 更像远程服务化接入。

这和你过去的经验一对照就很清楚了。前者类似本机 sidecar 或本地插件，延迟低、部署简单、边界更封闭；后者更像独立服务，可多客户端复用，适合企业平台化能力暴露。以后你做 Java 主系统 + Python Agent sidecar，或者 Java 内部能力暴露成远程 MCP Server，这里就是架构分界点。

第四个关键点，是会话和安全不是补充说明，而是协议设计的一部分。虽然今天这篇不展开授权章节，但仅从生命周期和总览里你也已经能看到几个很强的工程信号：

1. 版本要协商，不是默认一致。
2. 能力要协商，不是默认可用。
3. 会话有开始和结束，不是永远在线。
4. HTTP 场景下授权是正式章节，不是实现细节。

这说明 MCP 的方向并不是“写个 JSON 就能互通”，而是向真正可交付的协议演进。对你这种做过客户现场交付的人，这个味道应该很熟：真正能上线的协议，一定同时关心能力、边界、状态和安全。

如果把今天这两份文档翻译成你能直接拿来用的工程语言，可以记成一句话：

MCP 不是“模型调工具的语法糖”，而是“Agent 世界里的标准化接入会话”。

3 个关键收获：

1. MCP 先是协议会话，再是工具调用；`initialize -> initialized -> operation` 这条链路必须先立住。
2. 客户端和服务端都要声明能力，后续通信只能在已协商边界内进行。
3. MCP 的价值不只在本地工具接入，还在为远程服务化、权限控制和平台化复用打基础。

和 Agent 工程的关系：

你后面无论是看 OpenAI Agents SDK、自己写 MCP Server、还是用 Spring AI 把企业内部能力暴露出去，都会遇到同一类工程问题：谁持有会话、谁控制文件边界、哪些能力能暴露、失败后怎么重连、版本不兼容时怎么降级。今天这篇正是在补这些“不会出现在 Demo 首页，但一定会出现在真实系统里”的底层意识。它和你原来的系统集成经验不是断裂关系，而是直接迁移。

原文关键段落翻译（人工翻译，放在文末）：

1. 所有实现都必须支持基础协议和生命周期管理，其他组件可以按应用需要选择实现。这说明 MCP 的最低门槛不是“会列出 tools”，而是“会按协议建立会话”。
2. 初始化阶段必须是客户端和服务端的第一轮交互。客户端发送自己支持的协议版本、能力和实现信息；服务端返回自己的版本、能力和实现信息；完成后客户端再发送 `initialized`，表示进入正常操作。
3. 在正常运行阶段，通信双方必须遵守协商后的协议版本，并且只能使用那些已经成功协商过的能力。
4. MCP 既规定生命周期，也规定标准传输方式；核心目的不是增加复杂度，而是把上下文接入、工具暴露和会话边界变成可互操作的正式协议。
