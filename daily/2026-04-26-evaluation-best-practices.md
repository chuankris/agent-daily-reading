# 2026-04-26 Evaluation Best Practices

原文来源：

- OpenAI, Evaluation best practices
- 链接：https://platform.openai.com/docs/guides/evaluation-best-practices
- 来源类型：官方文档

为什么今天读这个：

你现在刚开始进入 Agent 学习，很容易把注意力全放在模型、框架和工具上。但真正让系统持续变好的，不是“又学了一个新名词”，而是有没有一套办法判断它到底有没有变好。

原文翻译（先读这一段）：

这篇文档强调，评测是一个持续循环：先定义目标，再建立数据集，再运行评测并分析结果，然后根据结果迭代系统。文档建议优先使用和真实用户请求接近的数据，不要只依赖演示样例。评测不应该只看最终答案，还应关注调用工具、检索质量、延迟和成本等系统行为。对于复杂 Agent，要按架构分层评估，避免只凭主观感受判断“看起来不错”。

中文精读：

这篇官方文档最关键的出发点是：生成式 AI 天生带有不稳定性。同一个输入，模型可能给出不同输出，所以传统软件里那种“写几个固定断言就结束”的测试方法，对 Agent 系统是不够的。

文档强调，评测首先要从目标开始，而不是从指标开始。正确顺序应该是：

1. 先定义你到底要验证什么
- 例如回答是否正确
- 例如工具调用是否准确
- 例如是否满足真实用户任务

2. 再收集数据集
- 生产日志
- 专家手工样本
- 历史失败案例
- 边界条件与对抗样本

3. 再定义指标
- 例如上下文召回
- 例如函数调用准确率
- 例如人工好评率

4. 再做版本比较和持续评估

这和你后面做 Agent 工程非常相关，因为 Agent 的不确定性来源比普通 LLM 调用更多：

1. 用户输入本身不稳定
2. 中间工具调用可能失败
3. 多步流程可能在中间偏航
4. 多 Agent 或多 handoff 可能产生循环和漂移

文档里还有几个很有价值的提醒：

1. 要避免 vibe-based evals
- 也就是“感觉好像能用了”
- 这是 Agent 项目里特别常见的陷阱

2. 要重视 human feedback
- 自动评分很重要
- 但必须和人类判断校准

3. LLM judge 更适合做比较、分类、打分
- 不要把它用成完全无约束的开放式评价器

4. 复杂度越高，越要把评测拆到架构层次去看
- single-turn
- workflow
- single-agent
- multi-agent

最适合你现在拿去用的一个思路是：

即便你现在只有一个最小 Agent 服务，也可以很早建立评测意识，比如给 `POST /task` 设计 10 条固定输入，记录：

- 是否正确返回
- 是否结构化输出合规
- 是否错误处理清晰
- 平均耗时是多少

这样你后面每次改 Prompt、改 Tool schema、改接口，都不是盲改。

你只需要记住的 3 个点：

1. Agent 迭代不能靠感觉，必须尽早建立评测集。
2. 评测顺序是：先目标，再数据，再指标，再持续运行。
3. 架构越复杂，评测越要分层，而不是只盯最终答案。

和 Agent 工程的关系：

这篇会帮助你从一开始就走一条更像工程师的路。你不是去“试玩 AI”，而是在学怎么把一个带不确定性的系统做成可比较、可回归、可迭代的产品。

今晚可动手：

给你未来的最小 Agent 服务先写一个 `eval-cases.md` 草稿，先列 5 个样例：

- 输入
- 预期输出
- 是否涉及工具
- 失败时你要看什么日志
